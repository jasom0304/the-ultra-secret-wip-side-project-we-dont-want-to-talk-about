### Prompt 1 - ClaudeDM: Claude as a Service payé en SATs

**Demande :** Système de workflows interconnectés pour un service "Claude as a Service" payé en SATs.

**Use case global :**
1. Utilisateurs zappent une/plusieurs npubs cibles → solde SATs par npub en DB
2. `/claudeDM <requête>` via DM Nostr → vérifie solde, longueur, mots interdits
3. Si OK → Claude API → réponse + tokens consommés
4. Conversion tokens → SATs via Coinbase API publique
5. Décrémente solde, répond avec infos complètes

**Micro-workflows prévus :**
1. `zap-balance-tracker` - Détecte zaps → incrémente solde npub en DB
2. `claudeDM-entry` - Point d'entrée `/claudeDM <req>`
3. `check-balance` - Vérifie solde >= seuil (40 SATs)
4. `check-request` - Vérifie longueur (~4000 chars) + mots interdits
5. `claude-request` - Appelle Claude API, récupère réponse + tokens
6. `sats-conversion` - Tokens → USD → SATs via Coinbase
7. `debit-balance` - Décrémente solde
8. `success-response` - DM final avec réponse + infos coût
9. `error-insufficient` - DM "pas assez de SATs"
10. `error-bad-request` - DM "trop long" ou "mots interdits"

**Décisions d'architecture :**
- Tables dynamiques préfixées `wf_` + ID workflow
- Npubs cibles configurables par workflow (liste, somme globale)
- Seuil configurable (ex: 40 SATs)
- Limite input ~4000 caractères (configurable)
- Source conversion configurable (Coinbase pour l'exemple)

**Action requise :** Créer handler `sqlite` pour CRUD sur tables `wf_*`

### Prompt 1 - Fix syntaxe Handlebars

**Contexte :** Le coût en SATs n'apparaissait pas dans les réponses ClaudeDM ("Cout: SATs" au lieu de "Cout: X SATs").

**Problème identifié :**
- Plusieurs workflows utilisaient la syntaxe Liquid `{{ value | default: x }}` au lieu de la syntaxe Handlebars `{{ default value x }}`
- Cette syntaxe n'est pas reconnue par le moteur de templates Handlebars

**Fichiers corrigés :**
- `claudeDM-process.yml.example` : `{{ default actions.get_new_balance.response.value 0 }}`
- `claudeDM-insufficient-balance.yml.example` : `{{ default actions.get_balance.response.value 0 }}`
- `dm-to-mongodb.yml` : `{{ default match.category 'general' }}`
- `dm-to-mongodb.yml.example` : idem
- `zulip-workflow-notification.yml` : `{{ default parent.error "No error message" }}` + `{{ truncate trigger.from 20 }}`
- `zulip-workflow-ok.yml.example` : `{{ default parent.error "" }}`

**Règle retenue :**
- Liquid : `{{ value | filter: arg }}`
- Handlebars : `{{ helper value arg }}`

**Commit :** `aacd5ad` - fix: use correct Handlebars helper syntax instead of Liquid pipe syntax

### Prompt 2 - Intégration Handlebars dans ExpressionEvaluator

**Problème :** Les helpers Handlebars (token_cost, default, at_least) ne fonctionnaient pas.

**Fix :** Modification de `renderTemplate()` pour utiliser `templateEngine.render()` au lieu du regex custom.

**Commit :** `f1f938a` - fix: integrate Handlebars template engine into workflow rendering

### Prompt 1 - Fix schéma workflow_state

**Problème :** Deux entrées de balance pour la même npub à cause de `UNIQUE(workflow_id, namespace, state_key)`.

Quand `zap-balance-tracker` incrémentait avec `workflow_id=zap-balance-tracker` et `claudeDM-balance-gate` lisait avec `workflow_id=unknown`, deux lignes séparées étaient créées.

**Solution :**
- Changer la contrainte UNIQUE en `(namespace, state_key)`
- `workflow_id` devient metadata (dernier workflow à avoir touché la valeur)
- Namespace fournit l'isolation entre différents types de données

**Modifications `database.ts` :**
- `UNIQUE(namespace, state_key)` au lieu de `UNIQUE(workflow_id, namespace, state_key)`
- `getState()` : ne filtre plus par workflow_id
- `setState()` : ON CONFLICT sur `(namespace, state_key)`
- `deleteState()` : ne filtre plus par workflow_id
- `listStates()` : ne filtre plus par workflow_id
